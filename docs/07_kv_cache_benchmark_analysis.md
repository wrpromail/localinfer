# Qwen 架构演进与 KV Cache 性能基准测试分析

本文档总结了在构建 `localinfer` 本地纯手写推理引擎过程中的核心发现，包括对 Qwen 现代架构的演进分析，以及对大语言模型推理生命线（KV Cache）的深度量化基准测试。

---

## 一、 架构演进分析：Qwen3 vs Qwen3.5

在从 Qwen3 到 Qwen3.5 的演进中，模型架构并未走向“更复杂”的堆料，而是进行了一次极端的**计算量与显存压缩的实验**，目的在于为超长上下文和端侧受限设备寻找最优的 Pareto 平衡点。

### 1. Qwen3 (0.6B) - 标准的稳健基石
Qwen3-0.6B 使用了非常标准的 **纯 Transformer 全注意力架构 (Full Attention)**：
- **位置编码 (RoPE)**：标准 100% 维度旋转（即 `head_dim` 的所有维度均参与旋转编码）。
- **注意力机制**：采用保守的 **GQA (Grouped Query Attention)** 比例（16 组 Query 对应 8 组 KV，比例 2:1）。
- **定位**：这是一种极其成熟且符合理论上限（Scaling Laws）的经典解法，确保了模型的基础推理与理解能力。

### 2. Qwen3.5 (0.8B) - 激进的稀疏化探索
随着大模型时代的演进，**超长序列 (Long-Context)** 成为了刚需，但标准全注意力的计算复杂度和缓存显存消耗会呈爆炸式增长。因此，Qwen3.5 采用了一种非常激进的**混合架构 (Hybrid Architecture)**：
- **引入 Mamba (线性注意力) 变体**：大幅减少需要计算两两关系的全局注意力需求。
- **极致的稀疏 GQA**：将 Query 与 KV 的比例压缩到了极端的 4:1（例如 8 组 Query 只共享 2 组 KV），最大化缩减 KV Cache 体积。
- **局部旋转 (Partial RoPE)**：全注意力层仅对前 25% (`partial_rotary_factor=0.25`) 的维度应用旋转位置编码。
- **定位**：这是一种极限优化尝试，在尽量不明显牺牲推理质量的前提下，通过极致稀疏化来榨干算力与显存上限。

---

## 二、 核心实测：大模型推理的“生命线”—— KV Cache 

为什么模型架构师（如 Qwen3.5 的设计者）要拼了命地使用 GQA 等手段压缩 Cache 体积？我们在本地手写引擎中，通过开启/关闭 KV Cache 进行了一组极具代表性的量化对比。

### 1. 基准测试数据 (20 Token 生成)
在纯 CPU 环境、`bfloat16` 精度下，针对提示词 `"人工智能的未来是"` 连续生成 20 个 Token 的表现对比：

| 指标 | 🟢 有 KV Cache (默认) | 🔴 无 KV Cache (全量重算) | 结论 |
| :--- | :--- | :--- | :--- |
| **Decode 总耗时** | **1.1518 s** | **3.3893 s** | 有缓存**提速约 3 倍** |
| **单 Token 平均耗时** | **57.59 ms** | 169.47 ms | 延迟显著降低 |
| **生成速度** | **17.36 tokens/s** | 5.90 tokens/s | |
| **峰值物理内存 (RAM)** | 4658.97 MB | 4654.11 MB | (短序列下) 基本持平 |

### 2. 深度量化解读：逐词耗时的“雪崩效应”
无缓存模式在总耗时上输了 3 倍，但这并非线性落后。大模型自回归生成的数学瓶颈在于：如果舍弃缓存，计算复杂度将不可逆地坠入 **$O(N^2)$ 深渊**。

通过逐字统计延迟，这一瓶颈彻底暴露：
- **带有 KV Cache (O(N) 复杂度)**：
  每次只计算最新 1 个字的投影并与过去所有积累的特征点积。无论到第几个字，**单字耗时稳如泰山地保持在 ~55 毫秒**（如第10字: `54.03ms`，第15字: `58.56ms`）。
- **无 KV Cache (O(N²) 复杂度)**：
  模型每吐出一个字，都要把之前所有的字再输入进去重新计算完整的注意力矩阵。单字耗时发生线性雪崩：
  - 第 2 个字: `77.20 ms`
  - 第 10 个字: `111.77 ms`
  - 第 15 个字: `195.61 ms`
  - 第 20 个字: `205.48 ms`

可以推断，当上下文长度达到 1,000 甚至 10,000 Token 时，无缓存模式的单字耗时将膨胀到不可接受的几十秒甚至分钟级。

### 3. “内存基本持平”的假象
在基准测试中，两者峰值内存都约为 4.6GB 且几乎无差异。这是因为：
1. **短文本假象**：在 20 个 Token 的极短序列下，KV Cache 真正占据的空间仅有几十 KB。系统监测到的 4.6GB 几乎 100% 是被模型固态参数（0.6B in bf16）和 PyTorch 底层执行器的运行时开销所吃掉的。
2. **长文本诅咒**：如果将上下文拉长到 32K 或更高，静态权重占用的内存不再是主角，长长的 KV Cache 将反客为主，呈现线性暴增。这正是 Qwen3.5 为了追求长上下文，不得不妥协引入极端架构的底层驱动力。
