# 核心挑战与技术深坑记录 (Core Challenges & Pitfalls)

在从零手写并组装 Qwen3.5-0.8B 混合架构推理引擎的过程中，我们遇到了一系列极其隐蔽且致命的技术挑战。这份文档专门用于记录我们在抛弃高级框架（Hugging Face）、深入最底层物理引擎时，所踩过的“雷区”。

---

## 1. 致命的缩放因子遗漏 (Scaling Factor Omission)
**现象描述**：
在最初拼装完成的 `08_generate.py` 中，模型在消化完 Prompt 后，不仅没有吐出正常的中文，甚至连标点符号都不正常，而是直接、毫无预兆地连续输出了 `---------------`。

**根本原因**：
在线性注意力 (Gated Delta Network) 的更新公式中，我们最初遗漏了一个极其关键的数学步骤：对 Query 矩阵进行缩放。
在传统的 Attention 中，我们会除以 $\sqrt{d_{head}}$ 来防止 Softmax 爆炸；而在 Delta Rule 中，同样必须对 $Q$ 施加 `scale = 1.0 / math.sqrt(head_dim)`！
由于遗漏了这个除法，导致输入的 Prompt 在经过 18 层 Mamba 预填充时，数值被放大了成千上万倍。当来到最后一层时，隐状态 (Hidden State) 已经彻底数值爆炸（趋近于 Infinity 或 NaN），从而迫使模型只能吐出概率分布崩塌后最无意义的符号（连字符 `-`）。

## 2. 状态空间模型原生的精度漂移 (Numerical Instability of SSM)
**现象描述**：
在我们紧急补充了 $Q$ 的缩放因子后，模型停止了 `---------` 的灾难性输出，但依然陷入了概率陷阱，输出了不断重复的 `peredperedpered`。

**根本原因**：
这揭示了混合架构（尤其是状态空间模型）的一个终极工程难题：**原生 Python 浮点运算的累计误差**。
在它的核心循环中，有一步是：`rnn_state = rnn_state * g_t.exp()`。
这里的指数运算极为敏感。在纯原生 PyTorch（不依赖底层 C++ / Triton 算子）的 `bfloat16` 甚至 `float32` 下，哪怕只有万分之一的精度截断，经过 18 层架构的 24 次循环以及长句子的累乘后，这个误差也会被指数级放大。
这就是为什么即便数学公式完全正确，纯 Python 实现依旧无法生成自然语言。工业界（如 vLLM 或官方实现）强制依赖底层的硬件级算子（将所有循环压入 GPU 内部的高速缓存 SRAM 中计算）来对抗这种精度漂移。

## 3. Prefill 速度的“死亡螺旋” (The Prefill Bottleneck)
**现象描述**：
当我们运行脚本时，即便只输入了一句简短的“人工智能的未来是”，控制台也会卡顿几十秒，提示 `正在逐词流式消化 Prompt...`。

**根本原因**：
传统的全注意力模型在处理 Prompt (Prefill阶段) 时，是一次性把所有词扔进去，GPU 并行算完所有的 $Q \cdot K^T$ 矩阵。
但由于 Mamba (Linear Attention) 的特性是依赖上一时刻的状态，原生 Python 无法实现复杂的**前缀关联扫描 (Parallel Associative Scan)** 算法，导致我们被迫写了一个 `for` 循环：一个词一个词地喂给模型，强行把原本可以极速并行的阶段变成了龟速的串行吞吐。这也说明了这种“RNN复兴”架构极其依赖底层工程算法的配合。

## 4. Gated Attention 的伪装
**现象描述**：
我们在手写全注意力 (Full Attention) 时，遇到了 `[2048]` 和 `[4096]` 维度无法相乘的报错。

**根本原因**：
官方文档或 config.json 中的线索极少（仅有一句 `"attn_output_gate": true`）。实际上它的 `q_proj` 矩阵大小是正常情况的两倍，因为它在投影 Query 的同时，平行投影出了一个同样大小的 Gate (门控特征)。最终的注意力结果必须乘以 `silu(Gate)`，这意味着我们在写推理代码时，必须手动将 $Q$ 切分为两半。这与市面上绝大多数开源模型的标准注意力写法大相径庭，极具迷惑性。

---

**总结**：从头手捏大模型，不仅是把公式翻译成代码，更是与硬件精度、内存布局和数值稳定性做最残酷的肉搏。上述四大深坑，恰恰是我们在这个项目中收获的最宝贵经验。
